# 🛡️ SIIA Technical Security & API Architecture

This document provides a deep dive into the security measures and architectural patterns implemented in the SIIA (Scalable Infrastructure for Academic Information) platform.

---

## 1. Authentication & Identity Management

### 🔑 Password Hashing (PBKDF2)
We never store plain-text passwords. The system uses the industry-standard **PBKDF2 (Password-Based Key Derivation Function 2)** with a SHA-256 hash.

- **Mechanism**: When a user registers or updates their password, the `werkzeug.security` utility generates a salted hash.
- **Salt**: A unique, random salt is automatically added to each password before hashing, protecting against rainbow table attacks.
- **Verification**: During login, the system hashes the provided password and compares it against the stored hash.

```python
# app/models/user.py
def set_password(self, password):
    self.password_hash = generate_password_hash(password)

def check_password(self, password):
    return check_password_hash(self.password_hash, password)
```

### 🎫 JWT (JSON Web Tokens)
SIIA uses **Claims-Based Authentication** via JWTs. Instead of server-side sessions, the client stores a cryptographically signed token.

- **Payload**: The token contains the User UUID and their Role (admin, professor, student).
- **Signing**: Tokens are signed using a server-side `JWT_SECRET_KEY` (min 32 bytes in production).
- **Expiration**: Tokens expire after 1 hour to minimize the window of opportunity for stolen tokens.
- **Storage**: The React client stores the token in `localStorage` and injects it into the `Authorization: Bearer <token>` header for every API request using an Axios Interceptor.

## 2. Authorization (RBAC)

We implement **Role-Based Access Control (RBAC)** at both the UI and API layers.

### 💂 Backend Decorators
Every sensitive endpoint is protected by a custom `@requires_role` decorator that inspects the JWT claims before allowing the logic to execute.

```python
@announcements_bp.route('/', methods=['POST'])
@jwt_required()
@requires_role('professor', 'admin')
def create_announcement():
    # Only admins or professors reach this line
    ...
```

### 🧱 Frontend Protection
- **Route Guards**: A `ProtectedRoute` component wraps admin pages, redirecting unauthorized users to the home or login page.
- **Conditional Rendering**: UI elements like "Delete" or "Edit" buttons are only rendered if the user's role permits the action.

## 3. API & Data Security

### 🧬 UUIDs for Primary Keys
The system uses **Version 4 UUIDs** (Universally Unique Identifiers) for users instead of auto-incrementing integers.
- **Benefit**: Prevents "Insecure Direct Object Reference" (IDOR) attacks where a user could guess another user's ID just by changing a number in the URL (e.g., `/api/users/1` -> `/api/users/2`).

### 🧼 Data Sanitization
To prevent **Cross-Site Scripting (XSS)**, all rich-text content generated by the Studio editors is passed through `DOMPurify` before being rendered in the browser. This strips out malicious `<script>` tags or `onerror` handlers.

### 🛡️ CSRF & CORS
- **CORS**: The Flask backend uses `flask-cors` with a strict whitelist. In production, it only accepts requests from the specific Vercel frontend domain.
- **Headers**: Secure headers are enforced to prevent the site from being embedded in malicious iframes.

---

## 4. Infrastructure Security

### ☁️ Database Isolation
The **Aiven PostgreSQL** instance is isolated. The connection string includes `sslmode=require`, ensuring all data transmitted between Vercel and the database is encrypted via TLS.

### 🔑 Environment Secrets
Sensitive data (API keys, DB credentials, JWT secrets) are **never** committed to Git.
- **Local**: Managed via `server/.env` (ignored by Git).
- **Production**: Managed via **Vercel Environment Variables** and **Secret Files**.
